# Semantic configuration and semantic layer

`data-agent-core` supports two semantic inputs:

1. **Hand-authored semantic config** (YAML/JSON) for NL->SQL runtime
2. **Generated semantic model** (JSON/YAML under `.state/semantics/`) for analytics workflows

## Hand-authored semantic config example
```yaml
project: retail_demo
tables:
  - name: sales_daily
    description: Daily revenue by store and category.
    preferred: true
columns:
  - table: sales_daily
    name: revenue
    description: Gross revenue in USD.
metrics:
  - name: revenue
    expression: SUM(revenue)
dimensions:
  - name: category
    description: Product category.
business_rules:
  - Prefer preferred views before raw tables.
examples:
  - question: What were top categories yesterday?
    sql: SELECT category, SUM(revenue) FROM sales_daily GROUP BY 1
```

## Generated semantic model
Generated by `analytics_build_semantic_layer` from dataset profile + schema.

Stored in:
- `.state/semantics/<dataset>.json`
- `.state/semantics/<dataset>.yaml`

Model includes:
- `metrics` with SQL expressions and confidence
- `dimensions` with column bindings and confidence
- `business_rules`
- `preferred_time_dimension`

## Validation behavior
Semantic generation is fail-closed:
- unknown columns are rejected
- unsafe or invalid metric SQL is rejected
- unknown metric/dimension references are rejected

## Recommended workflow
1. Ingest dataset
2. Profile dataset
3. Build semantic layer
4. Review/adjust generated semantic model
5. Run analysis plan
